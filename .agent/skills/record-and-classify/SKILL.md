# 记录内容并分类

当用户向你描述日常记录时（例如"今天发生了xxx"或"我想记录xxx"），按以下步骤处理：

## 步骤

1. **解析与拆分**
   - 识别记录的时间（如用户未提供，使用当前时间）
   - **智能拆分**：如果用户的输入包含多个不同领域或互不相关的主题（例如同时提到工作进展和生活感悟），**必须**将其拆分为多个独立的记录事项。
   - **检查相关素材**：主动扫描 `inbox/resources/` 目录。**注意时效性**：只关注与当前记录时间相近（例如当天）的素材文件夹。如果素材的时间戳（或文件夹名称中的日期）与今天相差太久，则**忽略**该素材。只有符合时间的素材才需读取分析。
   - 对拆分后的**每一个**事项，分别循环执行以下步骤：

2. **识别输入类型** (对每个事项)
   判断这条记录属于哪种类型：

   **A. 项目驱动型输入**
   - 特征：为了完成某个明确目标而进行的学习、调研、行动
   - 关键词：为了、用于、准备、xxx项目、下一步要...
   - 归档：`projects/[项目名]/` 目录

   **B. 纯粹沉淀型输入**
   - 特征：信息积累，没有明确的项目目的，但可能将来有用
   - 关键词：学到了、看到了、想到了、觉得有意思...
   - 归档：`areas/` 或 `knowledge/` 目录

   **如何判断？** 问用户："这条记录是为了某个具体项目，还是单纯的信息积累？"

3. **主线关联度检查 (Main Thread Relevance Check)** - **必须执行**
   - 询问自己和用户："这条记录跟当下的核心目标（人生主线）有多大关系？"
   - **评分**：给出 0-100 的关联度评分。
   - **过滤**：如果评分低于 60 分且属于"盲目跟风"或"虚假忙碌"（即非纯粹娱乐也非有效工作），建议用户放弃记录或标记为 #低价值。
   - 在生成的记录文件中，必须包含 `> 主线关联度: [分数]` 字段。

4. **确定分类** (对每个事项)
   根据输入类型 and 内容判断归属：

   **项目驱动型** → `projects/[项目名]/log.md` 追加记录

   **纯粹沉淀型** →
   - `areas/工作/` - 工作相关的信息
   - `areas/学习/` - 学习、技能提升相关
   - `areas/生活/` - 日常生活、人际关系
   - `areas/健康/` - 运动、饮食、睡眠、心理健康
   - `knowledge/` - 方法论、认知框架、个人原则
   - `inbox/` - 无法明确分类的内容

5. **生成文件名** (对每个事项)
   格式：`YYYY-MM-DD-简短描述.md`
   例如：`2024-12-16-项目复盘思考.md`

6. **创建文件** (对每个事项)
   使用以下结构：

   ```markdown
   # [标题]
   
   > 记录时间: YYYY-MM-DD HH:MM
   > 分类: [领域]
   > 主线关联度: [0-100]
   > 标签: #标签1 #标签2
   
   ## 内容
   [用户描述的内容，可适当整理格式]

   ### 相关素材 (如有)
   - [素材1名称](../../inbox/resources/xxx/file1.jpg)
   - [素材2及说明描述]
   
   ## 所思所想
   [用户的思考 and 感悟]
   
   ---
   *由AI整理于 {{TIMESTAMP}}*
   ```

7. **告知用户**
   汇总告知用户所有已创建的文件及其位置

## 注意事项

- **禁止合并**：如果输入包含多个不相关的领域（如工作和生活），必须拆分为多条独立的记录分别创建文件，不要强行归为一类
- 如果用户提到的是项目相关内容，询问是否需要创建新项目或归入现有项目
- 保持原始表达，不要过度修改用户的语言风格
